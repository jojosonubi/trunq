## What was learned tonight

1. **Supabase SQL linter false positives** ‚Äî DO blocks with named PL/pgSQL variables (`_count`, `_row_count`) get misread as "tables without RLS." Tagged DO blocks (`DO $migration_check$ ... $migration_check$`) avoid the trigger. Hardcoding values into the migration also avoids it. If the linter dialog appears, click Cancel ‚Äî never "Run and enable RLS" without verifying.

2. **Middleware `PUBLIC_PREFIXES` matters** ‚Äî Routes that auth via header (task-secret, share tokens, webhooks) get blocked by session-cookie middleware unless explicitly listed. This was a Phase 3 oversight caught tonight.

3. **Anonymous data access needs service-role explicitly** ‚Äî The delivery portal page route used `createClient()` (auth-scoped) and 404'd for incognito visitors after the new RLS policies. Fixed by switching to `createServiceClient()`. Pattern: any page or route accessed by unauthenticated users must use service-role for DB queries.

4. **`NEXT_PUBLIC_APP_URL` is critical for self-chaining workers.** Without it, `getBaseUrl()` falls back to `VERCEL_URL` which has deployment protection and isn't reliably reachable.
